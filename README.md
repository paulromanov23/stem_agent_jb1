# Stem Agent: Task-Driven Differentiation for Competitive Programming

## Introduction

The STEM Agent was first introduced by Shen & Shen (2026). They built an agent that tackles
pluripotency at the protocol and tool-binding layer — an undifferentiated core that
specialises into protocol handlers and tool bindings depending on interaction context. The
challenge of this project was more fundamental: can an agent differentiate at the level of *strategy
and self-conception*, not just connectivity?

This means starting from general prebuilt knowledge and learning to adapt strategically to
solve a specific class of tasks: discovering how problems in that class are typically
solved and rebuilding its own reasoning process accordingly.

To implement this, I adapted the Shen & Shen pipeline into the **PACE pipeline**, four
phases mapping directly onto the biological stem cell lifecycle:

1. **Perception (P):** The agent receives its first exposure to the environment — `P(m)`,
   where `m` is a profile set of unsolved examples. The agent only observes, never acts.

2. **Adaptation (A):** The agent differentiates by writing its own **genome** — a structured
   config encoding classification rules and solving strategies derived from the profile. This
   produces the *progenitor*: partially differentiated but not yet mature.
   Formally: `A(P(m)) = Progenitor(r)`, where `r` is the primary rule set.

3. **Commitment (C):** The agent iteratively refines the genome against a validation set.
   Proposals that improve performance are committed, those that don't are discarded via
   *apoptosis*. The genome is the only thing that changes — the base model and tools
   remain fixed. A plateau counter stops the loop once no further improvement is found.

4. **Execution (E):** The mature genome is frozen. The fully differentiated agent solves the
   target task class using the strategies it developed, without further self-modification.

I chose **competitive programming** as the domain — algorithm selection and solution
generation for problems from Codeforces, LeetCode and AtCoder. The choice was
deliberate: evaluation is unambiguous (a solution either passes all test cases or it doesn't),
the task class has genuine structure a stem agent can learn (problem types, constraint
signals, algorithm families), and it is directly relevant to JetBrains' work on IDE code
completion. A stem agent that learns to classify algorithmic problem types and adapt its
solving strategy is a conceptual ancestor of an IDE agent that recognises what class of
problem a developer is working on.

## Methods

**Genome structure.** The unit of differentiation is a `SolverGenome` — a JSON-serialisable
dataclass containing a `system_prompt` encoding strategic meta-knowledge, a
`classification_rules` list of explicit IF/THEN mappings from problem signals to algorithm
families, a `strategy_map` with step-by-step approaches per algorithm type, and a
`solving_order` defining the meta-process applied to every problem. The genome encodes
*how to think about a problem*, not just what to do.

**Perception phase.** The progenitor reads the profile split in two passes. Phase A samples
one third of the problems and forms hypotheses — algorithm families, constraint signals,
common pitfalls — without solving anything. Phase B verifies those hypotheses against the
remainder, producing a structured `AlgoProfile`. This is then translated into the
generation-1 genome by a separate LLM call that maps observed patterns to classification
rules and strategy entries. Diff proposals are generated by the model itself — given the
current genome and the list of failed problems, it outputs a targeted JSON patch addressing
the identified root cause.

**Commitment loop.** Starting from generation 1, the loop runs up to `MAX_GENERATIONS`
iterations: identify validation failures silently, diagnose root cause, propose a minimal
genome diff, score the candidate, commit or discard via apoptosis. The genome is saved at
every committed generation, making the differentiation history fully inspectable.

**Environment.** After two abandoned approaches — a GitHub PR review agent (no
unambiguous ground truth) and a Judge0-based sandbox (fragile API, rate limits) — I built a
custom execution environment using Python's `subprocess` module. Each solution runs in an
isolated process; output is compared character-for-character against expected values.

**Data.** LiveCodeBench Lite (Jain et al., 2024) was chosen for two reasons: real
competitive programming problems (non-trivial, closer to day-to-day coding tasks), and
direct contamination control — problems were published *after* GPT-4o's training cutoff,
making performance differences attributable to the agent's strategy, not retrieval. 300
problems were loaded with streaming, the final split was **30 / 50 / 50** (profile /
validation / test) with `seed=42`. The test set was never touched until final evaluation.

**Model.** GPT-4o was used as the base solver, and o4-mini for Perception and Commitment
reasoning phases. Temperature was set to `0.2` with `top_p=0.1` for all solver calls. For
o4-mini, parameters were kept at default, as constraining `top_p` on reasoning models
degrades the internal search process, as discussed below.

## Results

Consistent across 5 runs at the final configuration. The mature agent outperformed the
baseline by a mean of **+0.2 Pass@1** on the held-out test set.

| Run | Baseline (val) | Gen 1 (val) | Mature (val) | Baseline (test) | Mature (test) | Δ test |
|-----|---------------|-------------|--------------|-----------------|---------------|--------|
| 1 | 0.28 | 0.52 | 0.56 | 0.32 | 0.50 | +0.18 |
| 2 | 0.24 | 0.52 | 0.52 | 0.30 | 0.44 | +0.14 |
| 3 | 0.30 | 0.54 | 0.58 | 0.30 | 0.48 | +0.18 |
| 4 | 0.28 | 0.52 | 0.52 | 0.28 | 0.50 | +0.22 |
| 5 | 0.24 | 0.56 | 0.54 | 0.26 | 0.52 | +0.26 | 
| **Mean** | **0.27** | **0.53** | **0.54** | **0.29** | **0.49** | **+0.20** |

With 50 test problems, one standard deviation of Pass@1 is ~0.07. A mean delta of +0.2
is 2.85 standard deviations above zero — not attributable to sampling variance. On average,
the commitment loop committed at generation 2, with apoptosis correctly firing on all
subsequent proposals. Gen 1 (progenitor only) accounted for most of the total improvement, the committed loop added marginal and inconsistent gains, in several runs the mature genome matched generation 1 exactly, suggesting the loop found no improvable signal beyond the progenitor's initial differentiation.

## Discussion

**Temperature was load-bearing for the whole architecture.** At default temperature,
Pass@1 varied from 0.2 to 0.6 across runs for the *same genome*, making it impossible to
distinguish genuine improvement from noise. The commitment loop committed noise as
signal and degraded the genome from generation 2 onward. At `t=0.2 / top_p=0.1`,
evaluation became reliable enough for the loop to converge correctly. 

From this observations I can draw a deeper conclusion: self-improvement loop that uses its own evaluation as a commit signal
*requires* near-deterministic evaluation to function. Additionally, it was seen that low temperature hurts the baseline (it
commits to a naive approach and sticks with it) while helping the mature genome, which has
something productive to be deterministic about.

**The progenitor was more impactful than the commitment loop.** Gen 1, produced from reading 30 profile problems without solving any of them, accounted for nearly all of the total improvement, with the committed loop adding marginal gains. The mean Gen 1 test delta was approximately +0.24, leaving little room for the commitment loop to improve further. This suggests that for strategy-level differentiation, the quality of the initial observation pass dominates — a finding that points toward investing more in the perception phase design rather than running more committed generations.

**The agent found structure with very little.** Given only problem statements and difficulty
labels — no algorithm tags, no solution hints — the progenitor reliably identified dominant
algorithm families and produced classification rules that transferred to unseen problems.

**What failed.** Setting up a reliable evaluation environment took 2 of the 6 available days.
The PR review agent had no unambiguous ground truth, Judge0 was fragile at bulk eval
scale. 

Early experiments produced a misleading +0.23 validation spike at generation 2 that
evaporated on the test set. Diagnosing this as evaluation noise led directly to the
temperature finding. Looking back, that spike was the most useful failure of the project.

**What I would do with more time.** Make behaviour parameters self-tunable — temperature
proved to be the most impactful variable yet it is fixed, allowing the agent to tune its own
eval parameters during differentiation would be a direct extension of the PACE model. Add
tool use to the mature agent so it can run test cases incrementally and revise solutions,
bringing it closer to real IDE agent behaviour. Additionally, I would add a holdout generalisation check to the
commitment loop to guard against validation overfitting.

## Setup

```bash
git clone https://github.com/paulromanov23/stem_agent_jb1
cd stem_agent_jb1
pip install -r requirements.txt
cp .env.example .env    # add OPENAI_API_KEY
python dataset.py       # download and split LiveCodeBench
python main.py          # run full PACE pipeline (45–60 min)
```
---
Note: The progenitor and commitment phases rely on the model returning structured JSON.
Occasional malformed responses (~1 in 15 runs) will raise a `json.JSONDecodeError` and
terminate the run. If this happens, simply rerun `python main.py` — the dataset split is
fixed by seed so results will be comparable.
